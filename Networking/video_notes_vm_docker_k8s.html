Video 1: Virtual Machines

▶ NetworkChuck — you need to learn Virtual Machines RIGHT NOW!!

What Exactly is a Virtual Machine?

🏠 Simple Analogy: A Computer Inside a Computer A Virtual Machine (VM) is a fake computer created entirely by software, running inside your real computer. It behaves exactly like a separate, independent computer — it has its own operating system, its own desktop, its own files — but it's all an illusion running on your actual hardware.
Deep Explanation

Your physical computer has real hardware: a CPU, RAM, a hard drive. A VM is a software layer that pretends to be a set of hardware. The VM's operating system thinks it's running on real hardware, but actually a special software called a Hypervisor is intercepting every request and translating it to the real hardware underneath.

Think of it like this: your physical computer is a stage. The VM is a play performed on that stage. The actors (OS, apps) believe they're in a real house, but it's all set pieces managed by the director (hypervisor).

Key Properties of a VM:

  • Complete isolation: Each VM is a sealed box. If VM-1 gets a virus, VM-2 is completely untouched.
  • Has its own OS: You can run Windows inside a Linux machine, or vice versa.
  • Has allocated resources: You assign it RAM (e.g., 4 GB), CPU cores (e.g., 2), and storage (e.g., 50 GB) from your real machine.
  • Disposable: You can delete a VM and create a fresh one in minutes. Break it? No problem — just destroy it.

Why Should You Learn Virtual Machines?

NetworkChuck highlights 3 compelling reasons why VMs are essential:

1. Learning Ethical Hacking Safely

When you practice hacking (on platforms like TryHackMe or Hack The Box), you use tools that could damage your main OS. VMs create a safe sandbox — if something goes wrong, your real computer is untouched. Kali Linux (the hacker's OS) is specifically designed to run inside a VM.

2. Try Any Operating System Without Risk

Want to learn Linux but don't want to wipe your Windows? Just create a Linux VM! You can run Ubuntu, Kali, Fedora, or even macOS inside a VM without touching your main OS. It's like test-driving a car without buying it.

3. Break Things Without Consequences

VMs are the ultimate playground. You can:

  • Experiment with server configurations
  • Test dangerous scripts or malware (safely isolated)
  • Practice system administration without risking real infrastructure
  • Take snapshots — a "save point" you can revert to instantly if something breaks
⚡ Real-World Use Case Companies use VMs to run multiple servers on one physical machine. Instead of buying 10 physical servers (₹50 lakh+), they buy 1 powerful server and create 10 VMs on it. This saves massive costs on hardware, electricity, and space.

Hypervisors: The Magic Behind VMs

The Hypervisor is the software that creates and manages VMs. Without it, VMs cannot exist. There are two types:

Type 2 — Hosted Hypervisor (For Beginners)

Runs as an App on Your Existing OS

You install it like any other application on Windows/Mac/Linux. It then lets you create VMs inside it.

  • Examples: Oracle VirtualBox (free!), VMware Workstation, Parallels (Mac)
  • How it works: VM → Hypervisor App → Host OS → Hardware
  • Pros: Easy to set up, great for learning, no special hardware needed
  • Cons: Slower because requests must pass through the host OS first
🎯 NetworkChuck's Recommendation Start with VirtualBox — it's free, works on Windows/Mac/Linux, and is the easiest way to get your first VM running.
Type 1 — Bare-Metal Hypervisor (For Pros)

IS the Operating System (Installed Directly on Hardware)

There is no Windows or Linux underneath. The hypervisor itself is installed directly on the bare metal hardware. It has "godlike control" over all resources.

  • Examples: VMware ESXi, Microsoft Hyper-V, Proxmox VE (free!)
  • How it works: VM → Hypervisor → Hardware (no middleman OS!)
  • Pros: Much faster and more efficient; used by AWS, Azure, Google Cloud
  • Cons: Needs a dedicated machine; more complex to set up
💡 Why Faster? In Type 2, every VM request goes: VM → VirtualBox → Windows → Hardware (3 hops). In Type 1: VM → Proxmox → Hardware (2 hops). Fewer hops = faster.

Practical: Setting Up VMs with VirtualBox

NetworkChuck demonstrates creating Kali Linux and Ubuntu VMs using VirtualBox on Windows 10:

1
Download VirtualBox from virtualbox.org — it's free and cross-platform.
2
Download the ISO file of your desired OS (e.g., Kali Linux from kali.org, Ubuntu from ubuntu.com). An ISO is like a DVD image of the operating system.
3
Create a New VM in VirtualBox: Click "New" → Name it → Select the OS type → Allocate RAM (at least 2-4 GB) → Create a virtual hard disk (25-50 GB).
4
Mount the ISO: Go to VM Settings → Storage → Click empty disk → Choose the downloaded ISO file.
5
Start the VM and follow the OS installation wizard — exactly like installing an OS on a real computer.
6
Install Guest Additions (VirtualBox tools) for better resolution, shared clipboard, and shared folders between host and VM.

What is Proxmox VE?

🖥️ Simple Analogy: Turning an Old Laptop into a VM Factory Proxmox is a Type 1 (Bare-Metal) Hypervisor that you install directly on a computer — replacing its operating system entirely. That computer then becomes a dedicated VM server that you manage from any web browser on your network. It's like converting an old laptop into a professional data center — for free!
Why Proxmox over VirtualBox?
FeatureVirtualBox (Type 2)Proxmox (Type 1)
Runs onInside your Windows/Mac/Linux OSReplaces the OS entirely — IS the OS
PerformanceGood for learning (some overhead)Near-native speed (no host OS overhead)
ManagementDesktop GUI appWeb-based dashboard (access from any device!)
Best forRunning 1-2 VMs on your daily driverRunning 5-20+ VMs on a dedicated machine
PriceFreeFree (community edition)

Installing Proxmox: Step-by-Step

Prerequisites

  • A spare computer (even an old laptop works!)
  • A USB flash drive (8 GB minimum)
  • An Ethernet cable (Proxmox generally doesn't work with Wi-Fi during install)
  • CPU Virtualization enabled in BIOS (Intel VT-x / AMD-V)

Installation Steps

1
Download Proxmox ISO from proxmox.com/en/downloads — get the "Proxmox VE ISO Installer".
2
Create a bootable USB using Rufus (Windows) or Ventoy (any OS). Select the Proxmox ISO and flash it to the USB drive.
3
Boot from USB → Select "Install Proxmox VE" from the boot menu.
4
Accept EULA → Select target hard disk → Configure country/timezone/keyboard.
5
Set root password and email — this is your admin login for the web interface.
6
Configure networking — assign a static IP address, gateway, and DNS. Proxmox does NOT work well with DHCP. Example: IP 192.168.1.100, Gateway 192.168.1.1.
7
Complete install, remove USB, reboot.
8
Access the web UI from any browser: https://YOUR_IP:8006. Login with username root and your password.
⚠️ Important Post-Install Step After first login, disable the enterprise repository (which requires a paid subscription) and add the no-subscription repository. Then update the server. Without this, you won't be able to install/update packages.

Setting Up Kali Linux VM in Proxmox

1
Download Kali Linux ISO from kali.org — get the bare-metal installer (64-bit), NOT the pre-built VM images (those are for VirtualBox/VMware).
2
Verify SHA256 checksum to ensure the download isn't corrupted or tampered with.
3
Upload ISO to Proxmox: In web UI → Local storage → ISO Images → Upload.
4
Create VM: Click "Create VM" → Name it → Select uploaded Kali ISO → Allocate storage (50-100 GB), CPU cores (2-4), and RAM (4-8 GB).
5
Start VM and install Kali — follow the graphical installer for language, user, disk partitioning.
6
Optional: Set up XRDP for remote desktop access from other machines — better experience than Proxmox's built-in console.
💡 Pro Tip from NetworkChuck The "complete offline" Kali installer includes hundreds of pre-installed hacking tools (Nmap, Burp Suite, Wireshark, Metasploit, etc.). Choose this if you want everything ready out of the box. The smaller "net installer" downloads tools on-demand.

Video 3: Docker Containers 101

▶ NetworkChuck — you need to learn Docker RIGHT NOW!!

The Problem Docker Solves

😤 The "It Works On My Machine" Problem You write a Python app on your laptop. It works perfectly. You send it to your colleague — it crashes. Why? Different Python version, missing libraries, wrong OS settings. Docker eliminates this entirely by packaging your app + ALL its dependencies into a single, portable container.
The Old Way (VMs) vs The Docker Way

We learned that VMs solved the "one server, one app" problem. But VMs have a massive drawback:

  • Each VM needs a full Guest OS (2-10 GB in size)
  • VMs take minutes to boot
  • Running 10 VMs = 10 copies of an operating system = huge waste of resources
  • Resources allocated to a VM are reserved even when idle — if you give a VM 8 GB RAM, that 8 GB is locked away even if the app uses only 500 MB

Docker's insight: What if we could get the isolation benefits of VMs without the weight of a full OS?

VMs vs Containers: The Critical Difference

What Gets Virtualized
AspectVirtual MachineDocker Container
VirtualizesThe entire hardwareOnly the operating system
OSEach VM has a full Guest OSContainers share the host's OS kernel
SizeGigabytes (2-10 GB per VM)Megabytes (50-200 MB per container)
Boot timeMinutesUnder 1 second
IsolationComplete (own OS, own kernel)Process-level (shared kernel, separate filesystem)
Resource usageReserved upfront (wastes idle resources)Uses only what it needs
Best forRunning different OS types, full isolationRunning many instances of same-OS apps efficiently
🔑 The Key Insight (Explained Deeply)

VMs virtualize hardware — each VM gets a fake CPU, fake RAM, fake disk. The hypervisor translates fake hardware calls to real hardware. Because each VM has its own OS kernel, a Linux VM can run on a Windows host.

Containers virtualize the OS — all containers on a machine share the same OS kernel (the core of the operating system). Each container gets its own isolated filesystem, processes, and network — but they all use the host's kernel to talk to hardware. This is why containers are so much lighter and faster.

Trade-off: A Linux container can only run on a Linux host (same kernel). If you need to run Windows apps on a Linux server, you still need a VM.

Docker's Core Vocabulary

The 4 Terms You Must Know
TermWhat It IsAnalogy
Dockerfile A text file with step-by-step instructions to build an image. It's code that describes your app's environment. A recipe — lists every ingredient and cooking step
Docker Image A read-only template built from the Dockerfile. Contains your code + all dependencies + config. Immutable. A frozen meal packaged from the recipe — ready to heat and serve
Docker Container A running instance of an Image. This is your live application. You can run many containers from one image. The meal being served — you can serve 100 plates from one recipe
Docker Hub An online registry where people share pre-built images (like GitHub for Docker images). A recipe book library — download tested recipes for free

Example Dockerfile Explained Line by Line

# Start from an official Python base image (pre-built by Docker Hub)
FROM python:3.11-slim

# Set /app as the working directory inside the container
WORKDIR /app

# Copy all files from your project folder into the container
COPY . .

# Install Python packages listed in requirements.txt
RUN pip install -r requirements.txt

# When the container starts, run this command
CMD ["python", "app.py"]

This Dockerfile says: "Take a clean Python 3.11 environment, put my code in it, install my dependencies, and run my app." The resulting container will run identically on ANY machine with Docker installed — your laptop, your colleague's Mac, or a server in Singapore.

Essential Docker Commands

# Pull an image from Docker Hub
docker pull centos

# Run a container from an image (interactive terminal)
docker run -it centos bash

# List all running containers
docker ps

# List ALL containers (including stopped ones)
docker ps -a

# Connect to a running container
docker exec -it <container_id> bash

# Build an image from a Dockerfile
docker build -t my-app .

# Run a container in detached (background) mode
docker run -d -p 8080:80 my-app

# Stop a container
docker stop <container_id>

# Remove a container
docker rm <container_id>
🌐 Docker Networking Basics

Each container gets its own IP address. Containers can communicate with each other. Docker uses a "bridge" network by default — all containers on the same bridge can talk to each other, but are isolated from outside traffic unless you explicitly expose ports with -p.

Video 4: Kubernetes (K8s)

▶ NetworkChuck — you need to learn Kubernetes RIGHT NOW!!

Why Do We Need Kubernetes?

📦 The Problem with Docker Alone Docker is amazing for running one container on one machine. But what happens when:
  • Your app gets popular and you need 50 containers across 10 servers?
  • A container crashes at 3 AM — who restarts it?
  • You need to push a new version without downtime?
  • Traffic spikes during a sale — you need to auto-scale from 5 to 50 containers?
Managing this manually is impossible. Kubernetes automates all of it.
Simple Analogy

🎼 The Orchestra Conductor

Docker containers are individual musicians. Each can play beautifully on their own. But with 200 musicians, you need a conductor to coordinate them — telling each one when to play, adjusting volume, replacing anyone who faints, and adding more musicians if the concert hall fills up. Kubernetes is that conductor.

🎯 Key Takeaway Docker = running containers. Kubernetes = managing containers at scale. Docker handles the "what" (package and run an app). Kubernetes handles the "where, when, and how many" (deploy, scale, and heal across many machines).

Kubernetes Architecture

The Two Types of Machines in K8s

Master Node (Control Plane) 🧠

The "brain" of the cluster. It makes all the decisions but never runs your actual app containers.

  • API Server: The "front door" — all commands go through here (from you or from kubectl)
  • etcd: A database that stores the desired state of the entire cluster (how many containers, which images, etc.)
  • Scheduler: Decides which worker node should run a new container (based on available resources)
  • Controller Manager: Constantly watches the cluster. If a container dies, it notices and tells the scheduler to create a replacement

Worker Nodes (The Muscle) 💪

These are the machines that actually run your containers. A cluster can have dozens or hundreds of worker nodes.

  • kubelet: An agent running on each worker that receives instructions from the master and ensures containers are running correctly
  • Container Runtime: The software that actually runs containers (usually Docker or containerd)
  • kube-proxy: Manages networking — ensures containers can communicate with each other and with the outside world

Core Kubernetes Concepts

1. Pod — The Smallest Unit

What is a Pod? A Pod is a wrapper around one or more containers. It's the smallest thing Kubernetes manages. In most cases, 1 Pod = 1 Container. Think of a Pod as a cubicle in an office — it holds one worker (container) and gives them a desk, phone, and network connection.
  • Each Pod gets its own IP address
  • Pods are ephemeral (temporary) — they can be created and destroyed at any time
  • If a Pod dies, Kubernetes creates a brand new Pod to replace it (not revive the old one)

2. Deployment — Managing Groups of Pods

What is a Deployment? A Deployment tells Kubernetes: "I want 3 copies of this Pod running at all times." Kubernetes then ensures exactly 3 Pods are always alive. If one crashes, it automatically creates a replacement. If you want 10 instead, just update the number — K8s handles the rest.
  • Scaling: Change replicas: 3 to replicas: 10 and K8s instantly creates 7 more Pods
  • Rolling Updates: When you deploy a new version, K8s replaces Pods one at a time. Old Pod is killed only after the new one is healthy. Users experience zero downtime.
  • Rollback: If the new version is broken, K8s can instantly roll back to the previous version

3. Service — Exposing Pods to the World

The Problem Services Solve Pods are temporary — they get created and destroyed, each time with a new IP address. How do users or other apps find them? A Service provides a stable, permanent IP/DNS name that always routes to healthy Pods, no matter how many come and go.
  • ClusterIP: Internal access only — for communication between microservices inside the cluster
  • NodePort: Opens a port on every worker node — basic external access
  • LoadBalancer: Creates an external cloud load balancer that distributes traffic across all Pods — this is what you use for production websites

Essential kubectl Commands

kubectl (pronounced "kube-cuddle" or "kube-control") is the command-line tool for interacting with Kubernetes:

# Check cluster connectivity and see all nodes
kubectl get nodes

# Create resources from a YAML file
kubectl apply -f deployment.yaml

# List all pods in the default namespace
kubectl get pods

# Get detailed info about a pod
kubectl describe pod <pod-name>

# Scale a deployment (change number of replicas)
kubectl scale deployment my-app --replicas=5

# Watch pods in real-time
kubectl get pods -w

# Expose a deployment as a LoadBalancer service
kubectl expose deployment my-app --type=LoadBalancer --port=80

# Delete a pod (K8s will auto-create a replacement!)
kubectl delete pod <pod-name>

# View logs from a pod
kubectl logs <pod-name>

Example: Deploying a Website with Kubernetes (YAML)

# deployment.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: coffee-website
spec:
  replicas: 3          # Run 3 identical copies
  selector:
    matchLabels:
      app: coffee
  template:
    metadata:
      labels:
        app: coffee
    spec:
      containers:
      - name: coffee-container
        image: nginx    # Use the Nginx web server image
        ports:
        - containerPort: 80

This YAML tells Kubernetes: "Always keep 3 Pods running the Nginx web server. If any die, replace them automatically."
